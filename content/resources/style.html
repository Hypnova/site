Style

Our opinionated programming techniques that may or may not enhance your programming and debugging.

Variable Naming

We find it more convenient to break some language specific naming convention. Here's ours.

If a problem gives a constant variable, let's say \(N\), then we use that variable in our program. int N;

Let's say we are given an array of \(N\) elements. For loop variables, we will use the variable \(n\), the lowercase version of the given variable. 

int[] array = new int[N];
for(int n = 0; n < N; n++) {
  array[n] = in.nextInt(); // Example usage
}

When given a constant lowercase variable (for example some old CCC problems), if possible, use a capitalized version in your code instead. This is so that your code is consistent across problems.

Grid problems are pretty common. Typically, variable pairs are \(M, N\) and \(R, C\). Loop variables should be named accordingly. Note that typically, \(R\) will be given first. Rows translates to \(y\) in a coordinate. The following is an example of input with a grid problem. 

for(int r = 0; r < R; r++) { // Rows first
  for(int c = 0; c < C; c++) { // Then columns
  // Read char or whatever is given
  }
}

For other variables, use a concise and descriptive variable name. You don't need the whole word, as long as you can understand it. Avoid exceeding 10 characters. We like to use a C++ and Java hybrid variable naming for multiple words: do not use underlines and do not use any capital letters. For example, dist, cur (usually used for current index, level, etc.), maxdiff (maximum difference duh!).

Variable Declaration

Where you declare your variables isn't too important. If you know for sure that you will use it in a method later, declare it globally. You could also declare it globally just to be safe. However, it may seem neater when declared within the main method.

In C, multidimensional arrays cannot be initialized with a variable size, thus, you must initialize them globally with constraints stated in the problem.

Methods

Use the same naming convention for methods (the C++ and Java hybrid; you won't be given methods in a problem). Make sure that parameter naming doesn't interfere with global variables.

When used well, methods can improve comprehension, however, with bad usage, it can clutter up your code. Use methods when you need to reuse a medium length snippet of code in multiple places. Use when it logically flows to use a method. Refrain from creating too many methods especially which aren't often used. Don't put all your code in a constructor and call it in the main method. In fact, never use a constructor (unless you're creating a custom class which you should also refrain from).

Of course, you must use methods when utilizing recursion.

Indentation

This is our preferred style, take it or leave it.

Use 4 spaces instead of tabs to indent. Spaces permit better compatibility.

We like to use a K&R variant. No spaces between a keyword or method amd parentheses nor between a variable and index brackets. Open brackets on the same line and closing brackets on a different line. Spaces used as necessary. Single line loops usually on the following line and single line if statements on the same line with no brackets for either. 

int main() {
int a[] = {1, 2, 3 * (1 + 1)};

for(int i = 0; i < 3; i++) {
  if(i == 1) {
    a[i + 1] = 0;
    // Do something else
  } else { // Same line
    a[0] = i;
    i = 0;
  }
}

for(int i = 0; i < 3; i++)
  a[i] = i;

  if(a[0] == 1) return 1;
}
